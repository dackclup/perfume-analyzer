## Working with the big HTML files

`index.html` is ~8.4k lines, `formulation.html` is ~7k. Each is a
single inline-script SPA (no build step, no bundler — but it does
load three small ES modules from `lib/` via
`<script type="module" src="lib/dom-utils.mjs?v=…">` BEFORE the
inline init activates, so `window.appUtils.escHtml` etc. are set in
time). To stay fast:

1. **Start at `.codemap.md`** — it lists every section banner and
   top-level function/const with line numbers. One Read of that file
   tells you exactly where to jump.
2. **Use `Read` with `offset` + `limit`**, not full-file reads. The
   code map gives you the line.
3. **`grep -n` with anchored prefixes** like `^function ` or
   `^const ` is cheaper than wide regex — most module-scope
   declarations sit at column 0.
4. **Use the section banners** — every major area starts with one of
   `// ===== Title =====`, `/* ===== Title ===== */`, or
   `// ── Title ──`. The codemap walks these. Add a banner when
   you create a new logical block so the next reader can find it.

## Order matters — TDZ caveat

Cache-bust v184/v187/v188/v189/v192 all fixed the same family of
bug: a `let`/`const` declared in a downstream feature block but read
by code that runs at the inline-script's first paint. When you add
a new top-level binding that init code might touch:

- **Hoist it to the top-of-script comment block** (look for the
  "Sort state" / "Compare-cart state" / "Render virtualization
  state" comments near line 1180). They live there for a reason.
- Init reads `_renderCancel`, `_resultSeq`, `_sortOrder`,
  `_compareCart`, `_RESULTS_LS_KEY`, `_RENDER_CHUNK_*`,
  `CAS_RE_SEARCH`, etc. If you add a similar `let`/`const`, check
  whether init touches it. If yes, hoist.
- The Round-1 `lib/dom-utils.mjs` helpers (`esc`/`escHtml`,
  `debounce`, `lsRead` …) ARE hoisted to the very top of the inert
  script via `const x = window.appUtils.x` re-bindings, so module
  consumers downstream don't TDZ on them.

## Cache-busting & versioning

The PWA service worker uses **network-first for HTML**, **stale-while-
revalidate with version-key match for `data/materials.json`**, and
**cache-first with `ignoreSearch` fallback for the bundled JS / ES
modules** (taxonomy, formulation_data, formulation_engine, lib/\*.mjs).

Both pages share a single version string of the form `2026-04-29-vNNN`.
**Do not bump it manually** — run:

```sh
npm run release    # bumps version.json + every cache-bust string +
                   # DATA_VERSION + sw.js CACHE_VERSION + sw.js SHELL_ASSETS
```

`scripts/release.mjs` is the single source of truth. It:

1. Reads `version.json`, increments the data version (`vNNN → vNNN+1`).
2. Replaces the version string in `index.html`, `formulation.html`,
   `data/materials.json`'s `meta.version`.
3. Recomputes the SHA-256 content hash of every `SHELL_FILES` entry
   and writes the new `CACHE_VERSION = 'perfume-shell-v3-<hash>'` into
   `sw.js`.
4. Regenerates the `SHELL_ASSETS = […]` array literal in `sw.js`
   between the `>>> SHELL_ASSETS_BEGIN` and `<<< SHELL_ASSETS_END`
   marker comments — so the precache list and the content hash can
   never drift apart.
5. Verifies that exactly one version string survives repo-wide and
   that the recorded hash matches the live content. Fails CI on drift.

`npm run release -- --check` does the verification step only (no
writes). Wired into `npm run lint:version` and the pre-commit hook.

If you add a new file the SW must serve offline, edit `SHELL_FILES`
in `scripts/release.mjs` and run `npm run release`. Do NOT edit the
`SHELL_ASSETS` list in `sw.js` directly — the next `release.mjs` run
will overwrite it.

## Conventions

- **String escapes** — DOM-bound user input goes through
  `escHtml()` (formulator) or `esc()` (analyzer alias for the same
  function). Both delegate to `lib/dom-utils.mjs`'s `escHtml` via
  `window.appUtils`. When you build a new template literal, escape
  every `${user-controlled}` substitution. There is one shared
  contract: `escHtml(null|undefined) === ''`, every other value is
  coerced to string and the standard five HTML entities are
  replaced (including `'` for attribute-quote safety).
- **localStorage** — read/write through
  `lib/storage.mjs` (`lsRead` / `lsWrite` / `lsRemove` /
  `lsGetString` / `lsSetString`) so corrupt JSON or private-mode
  exceptions never escape. `lsRead` accepts an optional shape
  validator for keys whose schema may evolve.
- **Defensive init** — every init call (renderResults,
  `_updateCompareCta`, `_restoreUrlState`, `_restoreResults`) is
  wrapped in `try/catch` so a single throw can't poison the rest of
  the script. `lib/dom-utils.mjs` exports a `safeInit(name, fn)`
  helper that does this with a tagged log.
- **Animation — first mount only** — cards animate via
  `style="--anim-delay:Xms"`; re-renders of the same CAS get
  `style="animation:none"` so nothing flickers. Look for
  `_renderedCas` set; preserve when touching the render path.
- **Wheel single-select** — clicking a slice/band clears every
  other slice/chip in that axis and lights only the latest pick.
  The chip drawer stays multi-select for power users.
- **No build step** — the site loads raw HTML + raw `lib/*.mjs`
  modules. ES-module `<script type="module">` is the only
  toolchain dependency. Don't add a bundler; the modules are
  small and load order is hand-tuned.

## Tests

```sh
npm install                # also installs the pre-commit hook (prepare lifecycle)
npm test                   # vitest run
npm run test:watch         # vitest watch
npm run lint               # eslint (HTML + tools/ + lib/)
npm run format             # prettier write
npm run format:check       # prettier check
npm run lint:data          # schema + cross-ref + ratchet (CI gate)
npm run lint:version       # release.mjs --check (CI gate)
npm run lint:blends        # legacy blends-only linter
npm run codemap            # regenerate .codemap.md
npm run codemap:check      # fails if .codemap.md is stale (CI gate)
npm run release            # bump version + propagate; see Cache-busting above
npm run setup              # re-install the pre-commit hook (idempotent)
```

Tests live under `tests/` and exercise `lib/utils.mjs` and
`lib/dom-utils.mjs`. The HTML inline scripts are not test-targeted —
they depend on a live DOM and the data layer. When you add a new
pure helper that's worth testing, put it in the matching `lib/`
module (`dom-utils` for DOM concerns, `material-shape` for
DB-row shape, `storage` for localStorage, `utils` for everything
else).

## Workflow tips for LLM editors

- **Bump the cache-buster** in the same commit that ships a behaviour
  change. Otherwise a returning PWA user will see a stale shell. Use
  `npm run release` — never edit version strings by hand.
- **Run `node tools/codemap.mjs`** after structural edits. The
  diff to `.codemap.md` is reviewable and tells future readers
  what moved. The pre-commit hook will refuse a commit with a stale
  codemap.
- **Use try/catch around every new init call.** A bare throw at
  the top level halts the rest of the inline script — see the TDZ
  series of fixes (v184/v187/v188/v189/v192). `safeInit()` from
  `lib/dom-utils.mjs` is the canonical wrapper.
- **Match the existing comment density** — long-running logic
  blocks have a multi-line preamble explaining _why_; short
  helpers have a one-line header explaining _what_.
- **Don't introduce a build step** without an explicit ask. The raw
  ES-module loading is the only toolchain — the rest is plain HTML.
- **Domain changes need a source.** IFRA caps, EU allergen list,
  Edwards taxonomy structure — every change must cite a published
  URL (IFRA standards page, EUR-Lex regulation, Edwards' published
  wheel). Audits flag unsourced changes as `[NEEDS EXPERT REVIEW]`
  and refuse to auto-fix.

## Where things live (cheat sheet)

| You want to …                | Look in                                                                             |
| ---------------------------- | ----------------------------------------------------------------------------------- |
| Change the search algorithm  | `index.html` Search Matchers section                                                |
| Tweak a filter axis          | `index.html` `_filterAxes` + `_AXIS_CHECKERS`                                       |
| Edit the fragrance wheel SVG | `index.html` `_buildWheelSvg` (analyzer) / `formulation.html` wheel renderer        |

| Add a regulatory token       | `index.html` `REGULATORY_VALUES` + `REGULATORY_LABELS`                              |
| Add a captive supplier       | `index.html` `SUPPLIER_BRANDS`                                                      |
| Edit IFRA caps               | `formulation_data.js` `IFRA_51_LIMITS` (cite IFRA Amendment URL in commit)          |
| Patch material data          | `data/materials.json` — use `scripts/add-material.mjs` for inserts; never hand-sort |
| Add an EU allergen           | `scripts/add-allergen.mjs --name "<common name>"` (cite Annex III source)           |
| Rename a taxonomy token      | `scripts/rename-family.mjs --kind subfamily\|facet --from <old> --to <new>`         |
| Audit DB mismatches          | `npm run lint:data` — schema + 20-axis cross-reference + ratchet                    |
| Add to SW precache           | Edit `SHELL_FILES` in `scripts/release.mjs`, then `npm run release`                 |

When in doubt: `grep -n "thingyou'relookingfor"` or open
`.codemap.md` and Ctrl-F.
